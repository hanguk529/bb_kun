# ZTNA 위협 시뮬레이션과 분석 워크플로우 — 무엇을, 왜 때리는가

> 이 글은 ZTNA 엔드포인트 에이전트 분석을 시작하기 전 **전체 위협 모델과 공격 시나리오의 지도**를 그리는 사전 문서다. 구체적인 도구 조작과 단계별 절차는 후속 글 "ZTNA 엔드포인트 에이전트, 어디를 어떻게 때릴 것인가"에서 다룬다.

---

## 들어가며

ZTNA 솔루션은 단일 바이너리가 아니다. 엔드포인트 에이전트, 컨트롤 플레인(관리 콘솔), 데이터 플레인(게이트웨이)으로 구성된 **분산 시스템**이며, 각 컴포넌트 사이에는 신뢰 경계가 존재한다. 공격자는 어느 한 점을 잡고 다른 점을 타격하거나, 경계 자체를 비틀어 권한·신원·트래픽을 위조한다.

따라서 분석에 들어가기 전에 두 가지를 먼저 확정해야 한다.

1. **어떤 방향의 공격 시나리오를 시뮬레이션할 것인가** — 공격 벡터의 4분면
2. **어떤 순서로 시스템을 해부할 것인가** — 분석 워크플로우 3페이즈

이 글은 그 두 가지를 정리한다.

---

## 파트 1. 위협 시뮬레이션 — 공격 벡터 4분면

ZTNA 인프라 전체를 놓고 보면 공격 방향은 네 가지로 갈린다. 각각이 노리는 자산과 파급력이 다르므로, 시작 전에 어느 시나리오를 목표로 할지 선택해야 한다.

### 1.1. Agent → Admin (Upstream Attack)

VM에 설치된 에이전트(Node 1)를 조작하여 **중앙 관리자 서버를 공격**하는 역방향 타격이다. 에이전트가 서버로 보내는 데이터는 '신뢰할 수 없는 입력값'으로 처리되어야 하지만, 많은 솔루션이 에이전트 데이터를 무조건 신뢰하는 결함을 가진다.

- **Telemetry 및 상태 데이터 변조**: 에이전트가 관리자 서버로 전송하는 기기 식별값(Hostname, MAC, OS 버전, 백신 상태 등)을 Frida나 API 후킹으로 변조한다. 이 입력값이 관리자 대시보드에 렌더링될 때 **Stored XSS**가 발생하거나, 백엔드 DB에 저장될 때 **Blind SQLi**가 발생하는지 확인.
- **역직렬화(Deserialization) 취약점**: 통신 프로토콜이 JSON/XML이 아닌 Java Serialized Object나 특정 바이너리 포맷이라면, 조작된 페이로드를 전송해 관리자 서버 인프라에서 **RCE**를 유발할 수 있는지 분석.

### 1.2. Admin → Agent (Downstream Attack)

관리자 콘솔의 기능을 악용하여 **연결된 모든 에이전트 기기에 악성 명령을 하달**하고 장악하는 시나리오다. 공급망 공격(Supply Chain Attack)과 유사한 파급력을 가진다.

- **업데이트 배포 메커니즘 악용**: 에이전트 버전 업데이트나 패치 푸시 기능에 **서명 검증(Signature Verification)** 로직이 있는지 분석. 검증이 누락되었거나 우회 가능하다면 커스텀 바이너리를 업로드하여 에이전트에서 SYSTEM/root 권한으로 실행시킬 수 있다.
- **정책 기반 명령 실행 (Policy Abuse)**: 특정 기기 상태에 따라 실행되는 **복구 스크립트(Remediation Script)** 나 커스텀 검사 항목을 통해 에이전트 OS 환경에서 **OS Command Injection**이 가능한지 검증.

### 1.3. 관리자 애플리케이션 → 호스팅 인프라 (AppSec Escapes)

ZTNA 솔루션 자체가 구동되는 **서버나 클라우드 환경을 탈취**하는 접근이다.

- **서버 측 요청 위조 (SSRF)**: Webhook 연동, Syslog/SIEM 로그 전달, 외부 IDP(SAML/OIDC) 메타데이터 가져오기 등에서 SSRF를 시도. 클라우드 환경(AWS/Azure/GCP)이라면 **메타데이터 API(169.254.169.254)** 접근으로 임시 자격 증명을 탈취하는 것이 1순위 목표.
- **설정 파일 파싱 결함**: 설정 백업/복원(Export/Import) 시 업로드되는 파일(XML, JSON, ZIP)에서 **XXE**나 **Zip Slip(Path Traversal)** 을 통한 임의 파일 덮어쓰기 → RCE.

### 1.4. 테넌트 격리 우회 (Cross-Tenant Isolation Bypass)

ZTNA 솔루션이 **멀티 테넌트 기반 SaaS**라면 가장 치명적인 벡터다.

- **BOLA/IDOR 테스트**: 관리자 API 요청에서 `tenant_id`, `org_id`, `policy_id` 등의 파라미터를 조작하여, 권한 없는 다른 고객사(기업)의 보안 정책을 열람하거나 에이전트 노드를 내 통제하에 편입시킬 수 있는지 검증. 관리자 계정 보유는 정상적인 API 구조와 파라미터를 모두 파악할 수 있다는 **강력한 이점**이다.

## 파트 2. 엔드포인트 시작 시나리오 — 핵심 공격 벡터 4가지

위 4분면 중 가장 현실적인 시작점은 **엔드포인트에서 출발하는 시나리오**다. 공격자가 일반 유저 권한의 VM을 한 대 잡았다고 가정하고, 거기서부터 타격해야 할 핵심 벡터 네 가지를 정리한다.

### 2.1. Local Privilege Escalation (LPE) 및 방어 기제 우회

ZTNA 에이전트는 기기 통제를 위해 **커널 드라이버나 SYSTEM/root 권한의 데몬**을 구동하며, UI는 일반 유저 권한으로 실행된다. 공격자는 일반 유저 권한을 탈취한 상태에서 에이전트를 매개체로 권한 상승을 시도한다.

- **IPC 취약점 타격**: UI 프로세스와 백그라운드 서비스 간 통신(Windows: Named Pipes, ALPC, RPC / Linux·Mac: Unix Domain Sockets, XPC)을 가로채고 조작한다. 권한이 낮은 UI가 데몬에 특정 파일 삭제·레지스트리 수정·프로세스 실행을 요청할 때 검증 로직이 부실하면 즉시 SYSTEM 권한 획득.
- **Arbitrary File Operation (임의 파일 조작)**: 에이전트가 로그를 기록하거나 업데이트 파일을 임시 폴더에 다운로드할 때 **심볼릭 링크/하드 링크 공격**을 통해 `C:\Windows\System32` 등 보호된 경로의 파일을 덮어쓰거나(DLL Hijacking) 삭제 가능한지 확인.

### 2.2. 인증 자산 탈취 및 비인가 기기 복제 (Device Identity Cloning)

ZTNA의 근간은 **'기기 식별'** 이다. 공격자는 설치된 에이전트에서 인증 크리덴셜을 추출하여, ZTNA가 설치되지 않은 자신의 해킹 PC(예: Kali Linux)를 '인가된 사내 기기'로 위장한다.

- **크리덴셜 및 인증서 추출**: 에이전트가 게이트웨이/컨트롤 플레인과 통신하기 위해 사용하는 **mTLS 클라이언트 인증서, JWT 토큰, 하드코딩된 API 키**를 메모리/디스크에서 추출. Windows의 경우 **DPAPI**로 보호되는 키를 리버싱으로 복호화할 수 있는지 확인.
- **통신 프로토콜 리버싱 및 Replay**: 추출한 인증 자산을 이용해 공격자 PC에서 커스텀 스크립트(Python 등)로 ZTNA 컨트롤러에 직접 연결. **정상 에이전트 없이도 사내망 접속 터널 세션을 맺을 수 있다면 크리티컬한 결함**이다.

### 2.3. 기기 상태 검증 조작 (Device Posture Spoofing)

ZTNA는 OS 버전, 특정 백신 프로세스 실행 여부, 레지스트리 키 존재 여부 등을 평가해 접근을 제어한다. 공격자는 이 측정 데이터를 조작해 보안 정책을 우회한다.

- **Telemetry Hooking**: Frida나 API 후킹으로 에이전트가 상태 정보를 수집하는 함수(예: Windows WMI 쿼리, 레지스트리 조회 API)의 반환값을 변조. 실제로는 백신이 꺼져 있고 구버전 OS임에도 컨트롤러에는 '정상 상태'로 보고되도록 조작하여 정책 검증을 무력화.

### 2.4. 컨트롤 플레인 및 데이터 플레인 직접 타격 (Upstream Attack)

엔드포인트에서 에이전트의 통신을 가로채거나, 앞서 만든 클론 기기로 ZTNA 인프라에 악의적인 페이로드를 직접 전송한다.

- **인프라 대상 Fuzzing 및 RCE**: 에이전트-게이트웨이 간 터널링 프로토콜이나 API 엔드포인트를 식별하고, **기형 패킷(Malformed Packet)** 이나 페이로드를 전송. 메모리 커럽션(Buffer Overflow), 역직렬화, SQLi(텔레메트리 데이터에 SQL 쿼리 삽입)를 통해 ZTNA 게이트웨이 자체 장악이 최종 목표.
- **터널 내 라우팅 우회 (Network SSRF)**: 에이전트는 정책상 허용된 내부망 IP(예: `10.0.0.5:80`)로만 접근해야 한다. 에이전트→게이트웨이 캡슐화 패킷을 조작하여 **정책에 없는 내부망 대역(예: `10.0.0.1:22` 관리자 SSH)으로 트래픽을 강제 라우팅**할 수 있는지 검증.

## 파트 3. 에이전트 구조 분석 — 무엇을 들여다볼 것인가

공격 벡터를 정했으면, 다음은 에이전트의 어떤 부분을 어떤 도구로 들여다볼지 확정해야 한다. 네 갈래로 정리한다.

### 3.1. 아키텍처 및 권한 분리 맵핑 (Process Mapping)

에이전트가 단일 프로세스인지 다중 프로세스인지 파악한다.

**도구**: Process Hacker, Sysinternals Suite (Procmon, Procexp)

**식별 대상**:
- SYSTEM/root 권한 백그라운드 서비스(데몬)와 일반 유저 권한 트레이/UI 프로세스의 **분리 여부**
- 네트워크 필터 드라이버(**WFP, NDIS**) 로드 여부
- 프로세스 간 통신 방식(Named Pipes, RPC, ALPC, Unix Domain Sockets 등) — **이 채널이 곧 LPE 타겟**

### 3.2. 바이너리 리버싱 및 방어 기제 분석 (Reverse Engineering)

동적 분석을 방해하는 요소를 제거하고 핵심 로직을 찾는다.

**도구**: IDA Pro, Ghidra, x64dbg

**분석 포인트**:
- 패킹(Packing) 및 난독화(Obfuscation) 적용 여부 → 언패킹
- 안티 디버깅(Anti-Debugging), 안티 덤프(Anti-Dump) 로직 식별 및 패치
- **기기 상태(Posture) 수집 함수 식별** (WMI 쿼리, 레지스트리 검사 API 호출 위치) → 추후 Frida 후킹으로 백신이 켜진 것처럼 속이는 지점

### 3.3. 크리덴셜 및 상태 저장 방식 분석 (Secret Management)

에이전트가 중앙 서버와 통신하기 위해 사용하는 인증 자산(인증서, 토큰, 키)의 생성·보관 방식을 뜯어본다.

**분석 포인트**:
- 하드코딩된 API 키나 암호화 키 존재 여부
- 동적 발급 mTLS 인증서·세션 토큰의 디스크 저장 경로
- Windows의 **DPAPI(CryptProtectData / CryptUnprotectData)** 로 자격 증명을 암호화해 레지스트리/파일에 저장하는지 → 복호화 가능하면 공격자 PC로 인증 자산 탈취 가능

### 3.4. 네트워크 통신 구조 식별 (Network Protocol Analysis)

에이전트-컨트롤 플레인-게이트웨이 트래픽을 평문으로 볼 수 있는 환경을 구축한다.

**분석 포인트**:
- TLS/SSL 통신 시 **인증서 피닝(Certificate Pinning)** 적용 여부 → 적용되어 있다면 리버싱을 통해 피닝 우회 Frida 스크립트 작성
- 프록시(Burp Suite)를 물려 통신 데이터 포맷(JSON, XML, Protobuf, 커스텀 바이너리) 확인
- **이 단계가 끝나야 SSRF·역직렬화·상태 데이터 변조 페이로드 주입이 가능해진다**

## 파트 4. 실전 분석 워크플로우 — 3페이즈

이론을 마치고 손을 댈 차례다. 효율적인 분석 워크플로우는 다음 3페이즈로 구성된다.

### Phase A. 동적 경계 식별 및 런타임 조작

**Sysinternals + Wireshark + Frida**

단순히 로그나 패킷을 보는 것을 넘어, **암호화되기 전의 메모리 병목(Choke Point)을 런타임에 낚아채는 것**이 핵심이다.

#### A-1. IPC 채널 및 파일 시스템 매핑 (Procmon)

- Procmon을 가동하고 에이전트를 **전체 생애주기**(로그인, 정책 업데이트 등) 동안 실행
- `\Device\NamedPipe\` 필터로 데몬(SYSTEM)과 UI(User) 간 통신 파이프 이름 확정
- `accesschk`로 접근 권한(RW)을 확인하여 타격 가능한 표적인지 검증

#### A-2. 통신 암호화 무력화 및 평문 트래픽 확보 (Frida + Wireshark)

- **SSL Pinning 우회**: Wireshark에서 TLS 핸드셰이크 실패/통신 단절이 보이면 피닝 적용. Frida로 `WinVerifyTrust`나 OpenSSL의 `SSL_CTX_set_custom_verify`를 후킹, 리턴값을 강제 조작(0 또는 참)하여 피닝 해제.
- **메모리 덤프 (프록시 우회)**: 커스텀 프로토콜이나 UDP 기반 통신이라 Burp Suite를 물릴 수 없다면 **네트워크 스니핑을 포기하고 암호화 직전 단계를 타격**. Frida로 `EncryptMessage`(SChannel)나 `SSL_write` API를 후킹하여 평문 페이로드를 실시간 덤프. 이를 통해 컨트롤 플레인으로 전송되는 **Device Posture의 정확한 JSON/Protobuf 구조** 확보.

### Phase B. 타겟팅된 정적 분석 및 런타임 검증

**Ghidra + x64dbg**

Ghidra로 전체 코드를 읽는 삽질을 멈추고, **'IPC 디스패처'와 '파서(Parser)'의 정확한 메모리 오프셋**을 찾는 데 집중한다.

#### B-1. 디스패처 역추적 및 메모리 중단점 설정

- **Ghidra**: Procmon에서 찾은 통신 파이프 이름이나 Frida로 확인한 암호화 API 호출부를 검색하여 해당 함수로 직행. `ReadFile`이나 패킷 수신 함수 이후의 명령어 분기문(Switch-Case)을 찾는다.
- **x64dbg**: Ghidra에서 찾아낸 분기문 시작 주소(Offset)에 **하드웨어 브레이크포인트(HW BP)** 설정. UI 프로세스에서 특정 액션(로그 수집, 업데이트 등)을 트리거했을 때 디버거가 해당 주소에서 멈추는지 확인 → 정적 분석 흐름의 런타임 검증.

#### B-2. 논리적 결함 및 메모리 취약점 식별

- 디버거로 레지스터와 스택을 확인하며 전달된 페이로드(파일 경로, 커맨드 인자) 처리 흐름 추적
- 입력값 검증 없이 `CreateProcess`, `ShellExecute`, `MoveFile` 등 민감한 시스템 API로 파라미터가 그대로 흘러 들어가는 흐름(**Command Injection, Directory Traversal**)을 찾아낸다

### Phase C. 익스플로잇 및 타격

**Custom Script + Frida**

분석된 구조를 바탕으로 SYSTEM 권한 데몬을 완전히 통제한다.

#### C-1. IPC 프로토콜 위조 및 LPE 타격

- Phase B에서 파악한 메시지 헤더와 페이로드 구조를 모방하여 **Python/C++로 공격용 클라이언트 제작**
- UI 프로세스를 종료하고, 공격용 클라이언트로 데몬 파이프에 직접 연결 → 조작된 페이로드(예: 취약한 파일 이동 명령어를 악용한 DLL Hijacking) 전송 → SYSTEM 권한 코드 실행

#### C-2. 런타임 페이로드 변조 (In-Memory Fuzzing)

파이프 통신마저 암호화되어 있거나 서명 검증이 있어 클라이언트 위조가 까다롭다면 다시 Frida를 꺼낸다.

- UI 프로세스에 Frida를 물려, UI가 **정상적인 명령 프레임을 생성하고 서명/암호화하기 직전의 함수**(Ghidra로 식별한 오프셋)를 후킹
- 전달되는 인자(Argument) 문자열을 악의적인 페이로드로 런타임에 덮어써버리면, **에이전트의 정상 로직이 알아서 암호화와 서명을 거쳐 데몬으로 전송**

## 마치며

ZTNA를 공격한다는 것은 결국 **분산된 신뢰 경계를 하나하나 무너뜨리는 작업**이다. 이 글에서 정리한 그림을 다시 압축하면 이렇다.

- **공격 방향 4분면** — Agent↔Admin, AppSec Escape, Cross-Tenant
- **엔드포인트 시작 핵심 벡터 4가지** — LPE, Identity Cloning, Posture Spoofing, Upstream
- **에이전트 분석 4갈래** — Process, Binary, Secret, Network
- **워크플로우 3페이즈** — 동적 경계 식별 → 타겟팅 정적 분석 → 익스플로잇

여기까지 위협 모델이 그려졌다면, 다음 글에서는 이 지도를 들고 실제 도구를 잡는다. Sysinternals부터 Frida까지 단계별로 키보드를 두드리는 절차는 **"ZTNA 엔드포인트 에이전트, 어디를 어떻게 때릴 것인가"** 에서 이어진다.
